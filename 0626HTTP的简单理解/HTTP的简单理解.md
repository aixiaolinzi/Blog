HTTP的简单理解
##前言
看一些公司的招聘信息的时候，经常看到熟悉HTTP，今天我就反问了一下我自己什么是HTTP，然后就一脸mengbi懵逼了。
##任务
算是HTTP相关信息的整理吧！尽可能的理解一下HTTP。
###HTTP的特性
- HTTP构建于TCP/IP协议之上（就是工作在应用层），默认端口号是80
- HTTP是无连接无状态的

###HTTP的请求报文
HTTP 协议是以 ASCII 码传输，建立在 TCP/IP 协议之上的应用层规范。规范把 HTTP 请求分为三个部分：状态行、请求头、消息主体。类似于下面这样：
```groovy
<method>  <request-URL>  <version>

    <headers>
    
    <entity-body>
```
###Http定义了与服务器交互的不同方法
最基本的4种，分别是***GET***，***POST***，***PUT***，***DELETE***。
**OPTIONS**
返回服务器针对特定资源所支持的HTTP请求方法。也可以利用向Web服务器发送'*'的请求来测试服务器的功能性。
**HEAD **
向服务器索要与GET请求相一致的响应，只不过响应体将不会被返回。这一方法可以在不必传输整个响应内容的情况下，就可以获取包含在响应消息头中的元信息。
**GET **
向特定的资源发出请求。注意：GET方法不应当被用于产生“副作用”的操作中，例如在Web Application中。其中一个原因是GET可能会被网络蜘蛛等随意访问。
GET用于信息获取，而且应该是`安全的` 和 `幂等的`。
所谓`安全的`意味着该操作用于获取信息而非修改信息。换句话说，GET 请求一般不应产生副作用。就是说，它仅仅是获取资源信息，就像数据库查询一样，不会修改，增加数据，不会影响资源的状态。<br>
`幂等的`意味着对同一URL的多个请求应该返回同样的结果。
**POST **
向指定资源提交数据进行处理请求（例如提交表单或者上传文件）。数据被包含在请求体中。POST请求可能会导致新的资源的建立和/或已有资源的修改。
**PUT **
向指定资源位置上传其最新内容。
**DELETE **
请求服务器删除Request-URI所标识的资源。
**TRACE **
回显服务器收到的请求，主要用于测试或诊断。
**CONNECT ***
HTTP/1.1协议中预留给能够将连接改为管道方式的代理服务器。
###HTTP的响应报文
HTTP响应与HTTP请求类似，HTTP响应也由3个部分构成，分别是状态行、响应头（Response Header）、响应正文。
状态行由协议版本、数字形式的状态代码、及相应的状态描述，各元素之间以空格分割。
常见的状态码有如下几种：
- `200 OK` 客户端请求成功
- `301 Moved  Permanently` 请求永久重定向
- `302 Moved Temporarily` 请求临时重定向
- `304 Not Modified` 文件未修改，可以直接使用缓存的文档
- `400 Bad Request` 由于客户端请求有语法错误，不能被服务器所理解。
- `401 Unauthorized`请求未经授权。这个状态代码必须和WWW-Anthenticate报头域一起使用
- `500 Internal Server Error` 服务器发生不可预期的错误，导致无法完成客户端的请求。
- `503 Service Unavailable`服务器当前不够处理客户端的请求，在一段时间后，服务器恢复正常。

```groovy
HTTP/1.1 200 OK

Server:Apache Tomcat/5.0.12
Date:Mon,6Oct2003 13:23:42 GMT
Content-Length:112

<html>...

```
